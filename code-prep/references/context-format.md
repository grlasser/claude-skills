# CONTEXT.md Output Format Reference

This reference describes the structure Claude should produce when generating a `CONTEXT.md`
companion file as part of the code-prep skill.

The goal of `CONTEXT.md` is to give Claude Code everything it needs to orient itself in under
30 seconds — without reading every source file from scratch. It is optimized for AI consumption,
not human readability.

---

## File Header

```markdown
# Code Context: <module or project name>
_Generated by code-prep. Do not edit manually — regenerate with `code-prep` if source changes._

**Scope**: <single file | module | project>
**Python version**: <if detectable>
**Total files**: N  |  **Total lines**: N
**Generated**: <date>
```

---

## Section 1 — Purpose and Responsibility

One paragraph describing what this code does and why it exists. Include:
- The problem it solves
- Who calls/uses it (if inferrable)
- What it does NOT do (scope boundaries)

If the code has a module docstring, start from that and expand it with inferences from the code.

---

## Section 2 — Architecture Overview

A plain-language description of the structure. For multi-file modules, describe:
- How files relate to each other (layering, pipeline stages, data flow)
- Key design patterns in use (e.g., factory, observer, strategy, dataclass-heavy)
- Notable abstractions (base classes, protocols, interfaces)

For a single file, describe the internal organization (class hierarchy, helper vs. public functions).

Use a short ASCII diagram only if it genuinely clarifies relationships. Example:

```
[config.py] ──loads──> [engine.py] ──produces──> [output.py]
                              │
                         [validators.py]
```

---

## Section 3 — Module / File Summaries

One entry per file. Format:

```markdown
### `relative/path/to/file.py` (N lines)

**Purpose**: One sentence.
**Exports**: List of public names (classes, functions, constants) — be specific.
**Key dependencies**: Internal imports only (external libs are noise here).
**State**: Does it hold module-level mutable state? What?
**Change sensitivity**: HIGH | MEDIUM | LOW — explain why.
```

Change sensitivity guidelines:
- HIGH: Many callers, complex state, critical path, hard to test
- MEDIUM: Some callers, moderate complexity
- LOW: Utility/helper, well-isolated, easy to test independently

---

## Section 4 — Class and Function Inventory

### Classes

For each class, one block:

```markdown
#### `ClassName` (file.py:line)
Inherits: `BaseClass`, `Mixin`
Role: What responsibility does this class own?
Key methods:
  - `__init__(param: Type)` — what it initializes, what state it holds
  - `method_name(args) -> ReturnType` — what it does, key side effects
  - `_private_method()` — note if called by multiple public methods (hidden coupling)
Gotchas: Anything surprising — lazy init, thread-safety concerns, mutable defaults, etc.
```

### Standalone Functions

For each top-level function:

```markdown
#### `function_name(params) -> ReturnType` (file.py:line)
Purpose: What it does in one sentence.
Inputs: Describe non-obvious parameters, valid ranges, expected types if unannotated.
Returns: What the return value represents.
Side effects: File I/O, network, mutation, logging, exceptions raised.
Called by: [list if knowable from call graph]
```

---

## Section 5 — Dependency Map

Two sub-sections:

### Internal dependencies
Which files import from which other files in this project. Use a simple table or list.
Flag circular imports if any.

### External dependencies
List third-party packages imported (not stdlib). Group by purpose if obvious.
Example:
- `numpy`, `pandas` — data processing
- `httpx`, `requests` — HTTP
- `pydantic` — validation/serialization

---

## Section 6 — Call Graph Summary

Describe the most important call chains. Focus on entry points and hot paths.
Format as numbered chains:

```
1. CLI entry: main() → load_config() → validate_config() → Engine.run()
2. Error path: Engine.run() → retry_loop() → handle_failure() → notify()
3. Data pipeline: ingest() → transform() → OutputWriter.write()
```

Skip trivial or obvious chains. Flag any recursive calls.

---

## Section 7 — Change Risk Map

This section is the most valuable for Claude Code planning. For each change type, note the blast radius:

```markdown
| Change | Files affected | Risk | Notes |
|--------|---------------|------|-------|
| Add a new field to `Config` | config.py, engine.py, tests/ | MEDIUM | engine.py reads config fields directly |
| Rename `process()` method | engine.py, main.py, 3 test files | HIGH | no interface abstraction |
| Add async support to `Writer` | output.py, engine.py | HIGH | engine calls Writer synchronously |
| Change logging format | utils/logging.py | LOW | isolated, no callers depend on format |
```

---

## Section 8 — Known Gaps and TODOs

List:
- Functions with no docstring and non-obvious behavior (flag these as "needs annotation")
- Any `# TODO` / `# FIXME` / `# HACK` comments found in source
- Functions that raise `NotImplementedError`
- Missing type annotations on public interfaces
- Dead code (functions/classes that appear uncalled within the project)

---

## Section 9 — Suggested Claude Code Session Priming

A ready-to-paste prompt block the user can copy to the top of a Claude Code session:

```markdown
> **Context loaded**: See CONTEXT.md for full architecture map.
> **Working in**: <module name>
> **Key constraint**: <most important thing not to break>
> **Change target**: <what the user is about to modify>
```

---

## Tone and Style Notes

- Write for an AI reader, not a human. Be precise and dense. Avoid filler phrases.
- Use exact names (`ClassName.method_name`) not vague references ("the main function").
- When uncertain about intent, say "appears to" or "likely" rather than stating as fact.
- Flag anything that looks like a code smell or hidden coupling — this is where bugs live.
- The Change Risk Map and Gotchas are the highest-value sections. Prioritize completeness there.
